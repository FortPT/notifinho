# v2.3.2 acceptance checklist

Complete this checklist on VM-09 before merge and on VM-04 before considering
the production rollout complete.

## Automated release gate

- [ ] The complete Python suite passes.
- [ ] Python compilation, JavaScript syntax, YAML parsing, Compose rendering,
      documentation links, and `git diff --check` pass.
- [ ] `/api/health` reports version `2.3.2`.
- [ ] The release candidate image starts with platform schema 6.
- [ ] Upgrade and rollback smoke tests retain the matched configuration/state
      set.

## HTTP and HTTPS sessions

- [ ] `http://192.168.0.164:18080/` accepts login.
- [ ] `https://notifinho.local.fortpt.com/` accepts a separate login.
- [ ] A browser containing both legacy `__Host-` and standard cookies uses the
      configured standard cookie in dual-access mode.
- [ ] Refreshing either endpoint keeps its own authenticated session.
- [ ] Sign out clears the current session and the other endpoint remains
      independent.

## Source identity and categories

- [ ] Xen Orchestra, Grafana, Portainer, Proxmox, QNAP, Synology, TrueNAS,
      UniFi Network/Protect/Drive, Zabbix, Supermicro, HPE iLO, Dell iDRAC, and
      Home Assistant show their packaged vendor icons.
- [ ] A custom source shows the Notifinho icon.
- [ ] The category selector offers Virtualization, Monitoring, Storage,
      Networking, Hardware, Automation, Containers, Security, and Generic.
- [ ] Changing a category survives a page refresh and container restart.
- [ ] An enabled All Sources route marks Dell iDRAC and other discovered
      sources Active.

## Safe source removal

- [ ] Active sources do not show Remove.
- [ ] Inactive sources show Remove only to administrators.
- [ ] Cancel leaves the source unchanged.
- [ ] Confirm hides the source and records a `source.remove` audit event.
- [ ] Historical delivery records remain available.
- [ ] The backend rejects removal while an enabled exact or wildcard route
      covers the source.

## Destination tests and restart

- [ ] A destination-card Send test reaches the configured channel.
- [ ] The test uses the selected destination name and Notifinho identity/icon,
      not Home Assistant branding.
- [ ] Restart is absent from Settings and present at the top right for an
      administrator.
- [ ] Restart is hidden from normal users.
- [ ] Cancel does nothing; confirm requires a reason, records the audit event,
      and the restart policy brings the container back.

## Direct NFS/SMB managed mounts

- [ ] Rendered Compose uses root plus `DAC_OVERRIDE`, `FOWNER`, and
      `SYS_ADMIN`, with `cap_drop: ALL` and `no-new-privileges:true`.
- [ ] Startup logs contain `Platform state initialized (schema 6)`.
- [ ] Configuration is readable and logs/state are writable.
- [ ] `mount.nfs` and `mount.cifs` exist.
- [ ] The UNAS NFS target
      `192.168.0.163:/var/nfs/shared/Backups/Notifinho` passes Test.
- [ ] The mounted NFS options include `nolock`; NFSv3 does not attempt to start
      `rpc.statd`.
- [ ] A manual backup completes and its manifest is present on UNAS.
- [ ] A scheduled backup completes and retention behaves as configured.
- [ ] The SMB path is tested when a representative SMB server is available.

## Publication and production

- [ ] GitHub Actions passes on the draft PR.
- [ ] The release issue and implementation issues are on Project 1 with
      version `v2.3.2` and the correct status.
- [ ] The PR is merged before tag `v2.3.2` is created.
- [ ] Docker Hub and GHCR expose both `2.3.2` and matching `latest` images.
- [ ] VM-04 is pinned to `fortpt/notifinho:2.3.2`.
- [ ] HTTP, HTTPS, SMTP, Redfish/Home Assistant inputs, destination routing, and
      direct NFS backup remain healthy after the production restart.
