# Notifinho v2.2.0 release notes

22 July 2026

Notifinho v2.2.0 completes the operational WebUI started in v2.1.0. It keeps
`config.yaml` as the single configuration authority while adding persistent
notices, history-aware metrics, complete route-flow status, application and
input lifecycle controls, richer delivery records, profile pictures, health
checks, and scheduled private-state backups.

## Highlights

- Administrators publish information, warning, or severe notices. Ordinary
  notices are dismissed per user and do not return for that user.
- Configuration, route, scheduled-backup, and advertised-update notices cannot
  be dismissed. They resolve automatically when the associated condition is
  healthy or the advertised version is installed.
- Overview shows active Sources, Destinations, Routes, Applications, Success,
  and Requests for 10 minutes, 1 hour, 1 day, 1 month, or 1 year.
- Routing Flow includes enabled, disabled, and unhealthy routes with semantic
  priority, readable filters, output identity, and optional channel labels.
- Destinations retain their shared setting, render packaged output icons, use
  connection/presentation fields, and preview the destination actually chosen.
- Applications track last use and request count and can be enabled, disabled,
  or deleted. Disabled issued tokens stop authenticating immediately.
- Delivery history stores device, event, integration, description, severity,
  status, attempt, and input without the former unrelated “Final” label.
- Audit includes safe SQLite, YAML, credential, route, backup-target, and recent
  delivery checks.
- Daily, weekly, and monthly state snapshots can be copied to a host-mounted
  NFS or SMB directory without adding mount privileges to the container.

## Fresh installation

Use `fortpt/notifinho:2.2.0` and persist configuration, logs, secrets, state,
and the optional external-backup bind mount. Create the writable directories:

```bash
mkdir -p config logs/emails secrets state external-backups
chmod 600 config/config.yaml
chmod 700 logs logs/emails secrets state external-backups
```

Run the container as the host UID/GID that owns those paths. Keep secure cookies
enabled and publish the WebUI only through trusted HTTPS. The first-start setup
token remains single-use and no default password exists.

## Upgrade from v2.1.0

1. Stop Notifinho and copy the complete configuration and state directories.
2. Deploy `fortpt/notifinho:2.2.0` with the same mounts.
3. Confirm version `2.2.0`, schema 5, SMTP and HTTP listeners, and WebUI HTTP
   status 200.
4. Sign in and verify that the two default operational messages appear as
   notices instead of static Overview configuration panels.
5. Check all five Overview ranges, complete Routing Flow, destination sharing,
   application status, delivery fields, health checks, user status, and input
   toggles.
6. Trigger one harmless real event and confirm exactly one successful routing
   and send sequence.

The schema migration creates a verified snapshot in `schema-backups` before
moving schema 4 to schema 5. YAML remains the resource authority and is not
duplicated into fallback cards.

## External NFS or SMB backups

Mount the share on the Docker host with the host's existing NFS or CIFS tools
and credential storage. Bind the resulting directory into the container:

```yaml
volumes:
  - /mnt/notifinho-backups:/notifinho/external-backups
```

In **Inputs & backups**, choose NFS or SMB and set the container path to
`/notifinho/external-backups`. The setting records presentation and validation
metadata; the copy mechanism is identical for both because the host owns the
network mount. Notifinho rejects the filesystem root, symlink targets, missing
directories, and invalid schedules.

## Compatibility boundary

- Existing SMTP, HTTP, Redfish, UniFi, Home Assistant, parser, formatter,
  output, and unified YAML contracts remain compatible.
- Existing numeric route priorities are read and presented as semantic names.
  New WebUI writes use `critical`, `high`, `normal`, `low`, or `lowest`.
- Channel names are optional presentation metadata and are never fetched from
  Discord or Teams webhooks, avoiding undocumented credential-dependent API
  calls. Operators can enter the label shown in Routing Flow.
- `NOTIFINHO_AVAILABLE_VERSION` is optional. When a greater semantic version is
  advertised, a persistent update notice appears until the running version
  catches up.

## Security model

- Notice creation, backup scheduling, input mutation, shared resource changes,
  and health administration remain authenticated and CSRF protected.
- Ordinary dismissals are server-side and scoped to the current account.
- Profile images must be PNG, JPEG, or WebP, match their declared signature,
  and be at most 256 KiB. They are stored only in the mode-0600 database.
- Application values, hashes, environment variables, secret files, destination
  credentials, remote response bodies, and backup content never enter the
  WebUI.
- External shares are mounted by the host. The production container remains
  non-root, read-only, capability-free, and without filesystem mount authority.

## Rollback

Schema 5 is not readable by v2.1.0. A complete rollback requires the offline
pre-upgrade configuration and state copies together:

1. Stop the v2.2.0 container.
2. Preserve the failed v2.2.0 directories for investigation.
3. Restore the complete pre-v2.2.0 configuration and state directories.
4. Pin `fortpt/notifinho:2.1.0` and start the stack.
5. Confirm version 2.1.0, schema 4, and one real delivery.

Changing only the image is not a schema-safe rollback.
