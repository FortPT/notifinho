# Notifinho v1.8.0 release notes

## Release summary

Notifinho v1.8.0 expands the infrastructure-notification engine with Portainer
Alerting, Proxmox VE, and Synology DSM sources while preserving existing YAML,
SMTP, UniFi, Discord, and Microsoft Teams behavior.

The validation level is intentionally visible:

| Source | Transport | v1.8 validation |
|---|---|---|
| Portainer Business Edition 2.42.0 | Alerting webhook | Real firing and resolved delivery validated |
| Proxmox VE | SMTP and versioned webhook | Synthetic fixtures; real system pending |
| Synology DSM | SMTP, JSON, and form webhook | Synthetic fixtures; real system pending |

Proxmox and Synology are released as fixture-validated candidates so users can
perform compatibility testing when hardware becomes available. Issues #13 and
#43 remain open for that work.

## Native endpoints

```text
POST /portainer/alerts
POST /proxmox/events
POST /synology/events
```

The endpoints use the existing disabled-by-default HTTP listener and global
`http.shared_secret`.

- Every endpoint accepts `X-Notifinho-Token`.
- Portainer and Synology also accept one `?token=` query value because their
  notification-channel interfaces may not support a custom header.
- Synology accepts JSON and bounded `application/x-www-form-urlencoded` fields.
- Duplicate query or Synology form fields are rejected.
- Request bodies remain size-bounded and HTTP access logging is suppressed.

Use query-token endpoints only over a trusted direct management network. Avoid
reverse proxies that record query strings.

## Portainer Alerting

Portainer Business Edition 2.42.0 was validated end to end with firing and
resolved Alertmanager-compatible events. Notifinho creates one normalized
notification per alert and provides dedicated Discord and Teams presentation.

The integration consumes Portainer notifications. It does not poll the API or
retain administrator credentials. Portainer email compatibility remains
pending.

See [Portainer integration](../portainer.md).

## Proxmox VE candidate

The Proxmox adapter covers backup/restore, replication, node, cluster,
availability, storage, security, guest, and system classifications. Synthetic
SMTP backup results and the versioned `notifinho.proxmox.v1` JSON contract were
validated through development Discord.

Real Proxmox email layouts and notification-template variables remain pending.
See [Proxmox integration](../proxmox.md).

## Synology DSM candidate

The Synology adapter covers system, storage, disk/SMART, backup, replication,
UPS/power, package, security, network, and availability events. It supports
SMTP and the versioned `notifinho.synology.v1` contract as JSON or flat form
fields. All three synthetic transports delivered development Discord cards.

Real DSM email layouts, custom-provider variables, and content type remain
pending. See [Synology integration](../synology.md).

## Compatibility

- Existing SMTP, STARTTLS, SMTP AUTH, and source routing remain compatible.
- Existing UniFi Network, Protect, and Drive endpoints are unchanged.
- Existing Xen Orchestra, Zabbix, QNAP, Grafana, TrueNAS, and generic parsing
  remain available.
- Discord and Microsoft Teams configuration remains compatible.
- HTTP input remains disabled by default.
- Existing installations do not need to configure the new sources.
- Private `config/config.yaml`, secrets, and webhook URLs are not replaced by
  the image.

## Upgrade from v1.7.0

1. Back up `config/config.yaml`, logs, the Portainer stack definition, and the
   current image tag.
2. Pull `fortpt/notifinho:1.8.0` or
   `ghcr.io/fortpt/notifinho:1.8.0`.
3. Keep production pinned to `1.7.0` until the v1.8 image passes a temporary
   release-candidate smoke test.
4. Recreate the container and confirm the application reports `1.8.0`.
5. Confirm the existing SMTP and UniFi sources still deliver normally.
6. If using Portainer, add `routing.portainer` and configure the private direct
   Alerting URL documented in the Portainer guide.
7. Add Proxmox or Synology routes only when testing those sources.
8. Validate `401` without authentication and `204` for an accepted synthetic
   request.
9. Watch logs and destination cards before updating the production stack.

Do not commit the private runtime configuration or expose the HTTP listener to
the internet.

## Portainer stack rollout

For the FortPT deployment:

1. duplicate or export the existing production stack as a rollback reference;
2. change only the image tag from `1.7.0` to `1.8.0`;
3. preserve ports, volumes, environment variables, secrets, and Nginx Proxy
   Manager settings;
4. redeploy the stack;
5. confirm version, listener startup, existing notification delivery, and
   Portainer firing/resolved delivery;
6. leave Proxmox and Synology routes disabled unless actively testing them.

## Rollback

Recreate the production container with `fortpt/notifinho:1.7.0` or the
equivalent GHCR image. Restore the previous Portainer stack definition if more
than the image tag changed.

The new `/portainer/alerts`, `/proxmox/events`, and `/synology/events` behavior
is not available in v1.7.0. Disable new notification targets during rollback.
Existing SMTP and UniFi configuration remains usable.

## Validation

- 430 automated tests passed.
- Portainer BE 2.42.0 firing and resolved events delivered through the native
  authenticated endpoint to development Discord.
- Synthetic Proxmox JSON and SMTP notifications delivered to development
  Discord with structured backup presentation.
- Synthetic Synology JSON, form-encoded, and SMTP notifications delivered to
  development Discord.
- Header and supported query-token authentication returned `401`/`204` as
  expected.
- Example YAML parsed successfully.
- No production credentials, webhook URLs, private payloads, or infrastructure
  identifiers were committed.
