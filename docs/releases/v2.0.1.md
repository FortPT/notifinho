# Notifinho v2.0.1 release notes

## Release summary

Notifinho v2.0.1 enables the HTTP listener, authenticated platform API,
persistent platform state, and same-origin WebUI by default when their
configuration switches are omitted. Fresh installations and compatible v1.x
or v2.0.0 upgrades therefore expose the self-service interface without a
manual account-management command.

The first administrator is created through a secure, image-managed setup
flow. Until an account exists, each application start emits a new
cryptographically random, short-lived, single-use setup token to container
output. The operator opens the WebUI through HTTPS, supplies that token, and
chooses the administrator username and password. Notifinho never ships a
shared default password and does not allow unauthenticated first-visitor
registration.

## Highlights

- HTTP, platform state, the authenticated API, and the WebUI default to
  enabled only when the corresponding setting is absent.
- Every explicit `enabled: false` setting remains authoritative.
- A 256-bit random setup token is stored only as a SHA-256 digest, expires
  after 30 minutes, is rate-limited, and is consumed after successful setup.
- Restarting before setup rotates the token and invalidates the previous one.
- Existing account databases skip bootstrap and never emit a setup token.
- Secure session cookies, CSRF validation, login lockout, password hashing,
  ownership rules, secret redaction, and audit boundaries remain in force.
- The implicit state directory is now
  `/notifinho/config/platform-state`, allowing legacy deployments that already
  persist `/notifinho/config` to retain platform state.
- The production Compose definition continues to use the dedicated writable
  `/notifinho/state` mount.
- If an older configuration omits `platform.enabled` and its configuration
  mount is not writable, Notifinho warns and preserves the legacy YAML
  notification pipeline. Explicit platform enablement continues to fail
  closed when state cannot be initialized.

## Fresh installation

1. Use the versioned `fortpt/notifinho:2.0.1` image and persist the documented
   configuration, log, secret, and state paths.
2. Put the HTTP listener behind an HTTPS reverse proxy before exposing it
   outside a trusted management network.
3. Start the container and retrieve the setup token from its output:

   ```bash
   docker logs notifinho 2>&1 | grep -A8 "First-run administrator setup"
   ```

4. Open the WebUI through its HTTPS URL. Enter the setup token and choose a
   unique administrator username and strong password.
5. Sign in normally. The token is consumed, the bootstrap endpoint is closed,
   and future restarts do not emit another token.

Do not paste the token into tickets, chat, screenshots, or persistent log
collectors. If it is exposed before use, restart the container to rotate it.

## Upgrade from v2.0.0 or v1.9.7

1. Back up the active Portainer stack or Compose definition, configuration,
   logs, TLS material, secrets, and the complete platform state directory.
2. Validate `fortpt/notifinho:2.0.1` on a development host before production.
3. Confirm that `/notifinho/state` is a private writable persistent mount. A
   legacy deployment without that mount can use the persisted
   `/notifinho/config/platform-state` fallback.
4. Change only the image tag to `2.0.1` and recreate the container.
5. Confirm the version banner, listeners, non-root/read-only hardening,
   STARTTLS if configured, and one real legacy notification delivery.
6. If the database already contains an account, sign in with that account;
   bootstrap is skipped. If no account exists, use the one-time token from
   container output to create the first administrator.

Existing YAML inputs, parsers, formatters, Discord and Teams targets, routes,
ports, TLS settings, and webhook behavior are unchanged. The upgrade never
rewrites the YAML configuration.

## Compatibility boundary

The new defaults apply only to omitted switches. Operators can retain an
API-only, legacy-only, or intentionally disabled deployment by setting the
relevant `http.enabled`, `api.enabled`, `platform.enabled`, or `webui.enabled`
value to `false` explicitly.

Platform routing remains additive to the YAML router. Enabling both routing
models for the same event can produce duplicate notifications, so migration
must remain deliberate. Portable exports continue to exclude credentials,
passwords, token values and hashes, session material, and destination secrets.

## Security model

- Bootstrap is available only while there are no users.
- The plaintext token exists only in process memory and container output.
- The state database contains only its SHA-256 digest and expiry metadata.
- Attempts are bounded per client; invalid, expired, restarted, or consumed
  tokens are rejected.
- Account creation validates the normal username and password policy, creates
  an administrator, establishes a secure session, and records the transition.
- Session cookies stay `Secure`, `HttpOnly`, and same-origin; state-changing
  authenticated API requests require CSRF validation.
- Reverse-proxy HTTPS, trusted-network access, private state permissions, and
  careful container-log handling remain operator responsibilities.

## Rollback

Before upgrading, preserve a complete pre-v2.0.1 state snapshot. v2.0.1
upgrades platform state to schema 3, while v2.0.0 understands schema 2.
Therefore, do not start v2.0.0 against a database that v2.0.1 has migrated.

For a platform-enabled rollback, stop writes, stop the container, restore the
entire pre-upgrade schema-2 state directory, change the image tag to `2.0.0`,
and recreate the container. Restore the state and its secret files as one
atomic set.

If only the legacy YAML pipeline is needed temporarily, explicitly disable
`platform.enabled`, `api.enabled`, and `webui.enabled`, preserve the schema-3
state directory untouched, and run the previously validated image. Return to
v2.0.1 before re-enabling platform state. Never delete, partially copy, or
manually downgrade the SQLite database.

## Release acceptance

- The complete Python suite passes with pinned Python 3.13 dependencies.
- Python compilation, WebUI JavaScript syntax, YAML, Compose, and whitespace
  validation pass.
- A production Docker image builds from the immutable base image.
- An isolated VM-09 runtime verifies default-on startup, schema 3, token
  generation, rotation, expiry/single-use behavior, first-admin creation,
  secure session cookies, bootstrap closure, and graceful shutdown.
- An existing-account runtime confirms that restarts do not re-enable or
  disclose bootstrap.
- Explicitly disabled settings remain disabled.
- The existing YAML SMTP/webhook pipeline starts and routes unchanged.
- Pull-request and main-branch CI pass before the signed `v2.0.1` tag is
  pushed.
- The tag workflow publishes matching Docker Hub and GHCR `2.0.1` and
  `latest` images and creates the GitHub Release from this document.
