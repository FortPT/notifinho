# Notifinho v1.9.0 release notes

## Release summary

Notifinho v1.9.0 completes the server-side event and configuration foundation
planned before the user-facing v2.0 release. It adds Redfish hardware event
ingestion, dedicated Supermicro/HPE/Dell adapters, Home Assistant events, and
source-scoped backend APIs while preserving the v1.8.1 pipeline and YAML
deployment model.

## New sources

- Standard Redfish Event Service envelopes at `/redfish/events`.
- Explicit `/redfish/supermicro`, `/redfish/hpe`, and `/redfish/dell` adapters.
- Conservative Supermicro BMC/IPMI, HPE iLO AlertMail, and Dell iDRAC delivered
  email compatibility.
- Versioned Home Assistant automation events at `/home-assistant/events`.
- Generic source-scoped application events at `/api/events`.

Hardware adapters are fixture-validated candidates. They do not poll hardware,
store BMC administrator credentials, or claim compatibility with every vendor
firmware generation. Representative live validation remains required.

## Backend foundation

When `api.enabled` is true, v1.9.0 exposes health, masked config, validation,
sanitized logs, formatter preview, test-send, and atomic config-update
endpoints. Tokens use environment variables, owner-only files, or SHA-256
digests. Roles and source scopes are enforced with timing-safe comparison and
per-token/client rate limits.

Configuration updates are validated, backed up, written mode `0600`, atomically
replaced, and reloaded. `<configured>` secret placeholders preserve current
values during a masked round trip. Listener bind/port and transport-level
settings should still be changed with a controlled container restart.

Full local browser sessions, user account CRUD, CSRF, user-owned destinations,
visual route management, and the WebUI remain v2.0 work.

## Upgrade from v1.8.1

1. Back up `config/config.yaml`, the Portainer stack, mounted secrets, and TLS
   certificates.
2. Run the full test and isolated RC scripts from the v1.9.0 transfer package.
3. Keep `api.enabled: false` for the first upgrade unless the API is being
   deliberately tested on an isolated trusted network.
4. Pull `fortpt/notifinho:1.9.0` or `ghcr.io/fortpt/notifinho:1.9.0` only after
   the release workflow succeeds.
5. Change only the image tag, preserve mounts/ports/routes, and recreate the
   container.
6. Confirm the banner, SMTP listener, HTTP listener, existing source routes,
   and one safe existing-source event.
7. Add v1.9 tokens and source routes incrementally. Begin with synthetic
   fixtures before configuring vendor systems.

No new v1.9 section is mandatory for an existing v1.8.1 configuration. The new
inputs and API remain unavailable until configured.

## Rollback

Pin the production image back to `fortpt/notifinho:1.8.1` (or the equivalent
GHCR image), restore the prior stack definition, and recreate the container.
Existing v1.8.1 keys remain compatible. Remove or leave disabled the new
`api`/`redfish` sections; v1.8.1 will not use them. Restore a timestamped config
backup only if configuration was modified through the v1.9 API.

## Validation gates

- Complete automated suite and cache-free Python compilation.
- Isolated container reports v1.9.0 and starts SMTP/HTTP without errors.
- Existing unauthenticated endpoints remain compatible where intentionally
  configured; scoped v1.9 endpoints reject missing/wrong tokens.
- Supermicro, HPE, Dell, Home Assistant, and generic application fixtures route
  through the existing formatter/output pipeline.
- Config masking, validation, backup permissions, atomic replace, logs, preview,
  rate limit, and audit behavior pass.
- Product badge URLs are reachable from the merged `main` branch before tag.
