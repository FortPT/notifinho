# Notifinho v2.1.0 release notes

Released 22 July 2026.

Notifinho v2.1.0 makes the mounted `config.yaml` the single configuration
authority for destinations, routes, legacy application tokens, and regional
presentation settings. It replaces the temporary v2.0.2 takeover/fallback
model and removes duplicated YAML fallback cards and route rows.

## Highlights

- Valid external YAML edits appear in the WebUI and runtime automatically.
- Administrator WebUI edits are validated, backed up, atomically written to
  the same mounted file, and reloaded.
- Invalid YAML is reported without replacing the last known-good runtime state.
- Overview shows every enabled source → route → destination path plus metrics
  and recent deliveries; Quick Start is removed.
- Existing YAML application-token metadata appears safely without credentials.
- Global English/Portuguese, IANA timezone, and 12/24-hour presentation settings
  apply to the WebUI and notification formatters.
- Administrators manage shared YAML destinations/routes and other users'
  passwords; users receive read/preview access, and the current account changes
  its own password only in Account security.
- Test delivery reports actual success/failure, HTTP status, and safe error code.
- Persisted `delivered` outcomes now count as successful Overview deliveries.

## Fresh installation

Use the versioned `fortpt/notifinho:2.1.0` image and persistent configuration,
logs, secrets, and state mounts. Set:

```yaml
platform:
  enabled: true
  state_dir: "/notifinho/state"
  configuration_model: "unified_yaml_v1"
  secure_cookies: true

webui:
  enabled: true
  language: "en-GB"

presentation:
  timezone: "Europe/Lisbon"
  time_format: "24"
```

Keep secure cookies behind HTTPS. First startup still uses the short-lived,
single-use setup token printed to container output.

## Upgrade from v2.0.2

1. Stop Notifinho and create an offline copy of the complete configuration and
   state mounts.
2. Deploy `fortpt/notifinho:2.1.0` with the same mounts.
3. Confirm version `2.1.0`, schema 4, both listeners, WebUI HTTP 200, and no
   restart loop.
4. Sign in and confirm one destination list and one route list, with no YAML
   fallback duplicates.
5. Confirm every enabled route appears in Overview's active signal flow.
6. Change a harmless destination display name in the WebUI and verify it in the
   host `config.yaml`; edit it back in the file and confirm the WebUI refresh.
7. Send one real non-destructive source event and confirm one destination
   delivery plus a `delivered` history row.

On first synchronization, Notifinho matches existing v2.0.2 imported rows to
their YAML resources, adopts any remaining database-only destinations/routes
into YAML, writes `platform.configuration_model: unified_yaml_v1`, and removes
the old `platform.routing_authority` marker. A configuration backup is created
before that rewrite.

## Compatibility boundary

Schema 4 adds stable configuration keys to the private SQLite runtime mirror.
Before applying it, Notifinho writes a mode-0600 SQLite snapshot below
`platform.state_dir/schema-backups`. Listener binding changes—ports, addresses,
TLS certificates—still require a container restart. Destination credentials,
routes, API token metadata, language, timezone, and time format synchronize
without one.

The schema-4 database is not readable by v2.0.2. A full image rollback requires
the pre-upgrade offline state and configuration backups.

## Security model

- Only administrators mutate the shared YAML-backed destinations, routes, and
  preferences; the API enforces this independently of hidden browser controls.
- Users can inspect and preview safe destination output but cannot run a real
  shared-credential test delivery.
- YAML application entries expose only name, role, sources, rate limit, enabled
  state, and whether the source is an environment variable, secret file, or
  SHA-256 digest.
- Recursive redaction covers nested credential objects without hiding safe
  token metadata.
- WebUI writes use complete-document validation, bounded backups, mode-0600
  temporary files, `fsync`, and atomic replacement.

## Rollback

1. Stop the v2.1.0 container.
2. Restore the complete pre-upgrade `/docker/notifinho/config` and
   `/docker/notifinho/state` directories.
3. Pin `fortpt/notifinho:2.0.2` and redeploy.
4. Confirm schema 3, WebUI HTTP 200, and one real source delivery.

Do not delete schema metadata or manually remove columns. The automatic schema
backup is an extra recovery aid, not a substitute for the offline mount copy.

## Release acceptance

The release gate requires the complete Python suite, syntax compilation, WebUI
JavaScript validation, public YAML/Compose validation, production-image build,
an isolated published-image runtime test, bidirectional file/UI synchronization,
invalid-YAML last-known-good behavior, one-delivery routing, restart persistence,
and graceful shutdown. Pull-request CI and merged-main CI must pass before tag
`v2.1.0`; Docker Hub and GHCR digests must match afterward.
