# Notifinho v1.9.2 release notes

## Release summary

Notifinho v1.9.2 is a backwards-compatible Home Assistant presentation patch
for the v1.9 event platform. It turns common integration errors into concise,
structured cards and adds optional configuration-driven equipment aliases.

The existing `notifinho.home_assistant.v1` endpoint, authentication, routing,
and payload contract remain unchanged.

## Added

- Optional endpoint aliases map an observed address to a site-local device
  name and, when needed, a service label.
- Optional component aliases map a Home Assistant component to a known device,
  endpoint, and service.
- Discord and Microsoft Teams Home Assistant cards include a separate Error
  field when a structured error code is present.

## Fixed

- Bare IPv4 addresses are extracted from raw system-log text into Endpoint.
- Tapo/Kasa TriggerLogs failures are reduced to a concise event summary while
  retaining the endpoint and useful error code.
- Internet Printing Protocol failures receive a concise summary and canonical
  service label.
- Internal method names and repeated diagnostic fragments are omitted from
  these cards.
- A service name is no longer repeated as Device when no actual device is
  known. Explicit devices, detected devices, and configured aliases still take
  priority.

## Optional Home Assistant aliases

No configuration migration is required. Aliases are used only when the
optional block is present:

```yaml
home_assistant:
  aliases:
    endpoints:
      "192.0.2.10":
        device: "Building hub"
    components:
      "homeassistant.components.ipp.coordinator":
        device: "Office printer"
        endpoint: "192.0.2.20"
```

Keep site-local names and addresses in the private production configuration.
Shared documentation and automations should use generic examples.

## Upgrade from v1.9.1

1. Back up the production configuration, mounted token files, and stack
   definition.
2. Wait for the v1.9.2 release workflow to publish and verify both registries.
3. Pull `fortpt/notifinho:1.9.2` or
   `ghcr.io/fortpt/notifinho:1.9.2`.
4. Change only the image tag and recreate the container, preserving mounts,
   ports, routes, and secrets.
5. Confirm the banner reports Notifinho 1.9.2, configured listeners are
   healthy, and unauthenticated HTTP requests remain rejected.
6. Send one authenticated Home Assistant validation event and verify its
   destination and structured card fields.

Token and password rotation is not required.

## Rollback

Pin the image back to `fortpt/notifinho:1.9.1` (or the equivalent GHCR image)
and recreate the container with unchanged configuration and secrets.

The optional alias section is ignored by v1.9.1, so rollback does not require
removing it.
