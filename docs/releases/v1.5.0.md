# Notifinho v1.5.0 release notes

## Release summary

Notifinho v1.5.0 adds native support for UniFi Network, UniFi Protect, and
UniFi Drive while preserving every existing SMTP source, output, formatter,
route, conditional route, and logging path from v1.4.0.

Network and Protect use direct JSON webhooks through a new native HTTP input.
Drive parses RFC822 notification email delivered to the existing SMTP listener.
All three sources use the shared notification model and can route independently
to Discord, Microsoft Teams, or both.

## UniFi Network webhooks

UniFi Network Alarm Manager can send JSON to:

```text
POST /unifi/network
```

Strong envelope validation prevents arbitrary JSON from being classified as
Network. Notifications normalize event, controller, category, vendor severity,
client, network or Wi-Fi, connected-device, duration, RSSI, and event-time
context. Vendor severity remains available in metadata, and routine client
disconnects are not promoted to critical without stronger evidence.

## UniFi Protect webhooks

UniFi Protect Alarm Manager can send JSON to:

```text
POST /unifi/protect
```

The parser supports empty or multiple conditions and triggers, seconds or
milliseconds timestamps, valid event links, and unexpected additional fields.
Cards use the actual trigger device when it is a human-readable camera name.
MAC addresses, UUIDs, placeholders, and opaque identifiers remain internal and
are omitted from visible bodies, Discord fields, Teams facts, and INFO logs.

Motion, person, vehicle, and doorbell events can be high-volume. Use narrow
Alarm Manager event and device scope.

## UniFi Drive delivered-email parsing

Drive notifications are detected using sender-domain, product-identity, and
event-vocabulary evidence. The parser prefers plain text and uses a sanitized
HTML fallback that excludes scripts, styles, tracking or branding images,
signatures, action URLs, unsubscribe text, and footer boilerplate.

Backup, storage, disk, encryption, and administrative events receive
provisional severity and category normalization. Valid action links remain
available as clickable card actions without appearing in operational text.

## Native HTTP listener

The HTTP listener runs alongside SMTP and is disabled by default:

```yaml
http:
  enabled: false
  host: "0.0.0.0"
  port: 8080
  max_body_bytes: 1048576
  shared_secret: ""
```

It accepts JSON POST requests only, bounds request bodies, isolates malformed
requests, and shuts down gracefully with SMTP. Supported responses include 204
for accepted notifications, 400 for malformed input, 401 for failed optional
authentication, 404 for unknown paths, 405 for unsupported methods, and 413
for oversized bodies.

When `shared_secret` is non-empty, clients must provide:

```text
X-Notifinho-Token: <secret>
```

Comparison is timing-safe. Restrict the listener with host and network firewall
rules, and use authenticated TLS termination before any non-local exposure.

## Discord and Microsoft Teams

Network, Protect, and Drive have dedicated Discord embeds and Microsoft Teams
Adaptive Cards. They include Grafana-style Unicode application and status icons
plus readable icon-prefixed field labels. Existing status colors, payload
limits, clickable links, FortPT Labs footer, and Notifinho version footer are
preserved.

## Routing

The sources route independently and may share a destination:

```yaml
outputs:
  discord:
    enabled: true
    unifi:
      webhook: "PASTE_UNIFI_DISCORD_WEBHOOK_HERE"

routing:
  unifi_network:
    outputs:
      - output: discord
        target: unifi
  unifi_protect:
    outputs:
      - output: discord
        target: unifi
  unifi_drive:
    outputs:
      - output: discord
        target: unifi
```

Separate targets and equivalent Teams routes are also supported. Dedicated
output configuration uses `webhook`, never `webhook_url`.

## Docker and deployment

The image now exposes:

- `8025/tcp` for SMTP;
- `8080/tcp` for HTTP webhooks.

Existing SMTP-only deployments remain compatible because SMTP stays on port
8025 and HTTP remains disabled by default. Publishing a host mapping for port
8080 is required only when Network or Protect webhooks are enabled.

## Upgrade from v1.4.0

1. Back up the private configuration and current container definition.
2. Pull or build the v1.5.0 image.
3. Compare the public `config/config.example.yaml` with the private
   configuration without replacing secrets or existing routes.
4. Leave `http.enabled: false` for SMTP-only deployments.
5. When enabling Network or Protect, add the HTTP port mapping, firewall
   restrictions, optional shared secret, and the required routing keys.
6. Add a `unifi_drive` route only when Drive email is externally delivered to
   the SMTP listener.
7. Start the updated container and verify SMTP plus any enabled HTTP endpoint.

## Rollback

Stop the v1.5.0 container, restore the saved v1.4.0 image and deployment
definition, and restore the configuration backup if it was changed. Remove the
HTTP host-port mapping when it is no longer needed. Existing SMTP data and logs
remain compatible when their mounted directories are preserved.

## Known limitation

Notifinho parses UniFi Drive email after delivery but does not poll IMAP,
Microsoft Graph, Gmail, or other mailbox providers and stores no mailbox
credentials. Drive requires an external forwarding rule, SMTP relay, or other
delivery mechanism to reach the existing SMTP listener.
