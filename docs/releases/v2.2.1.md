# Notifinho v2.2.1 release notes

22 July 2026

Notifinho v2.2.1 is a focused hotfix for the route status control and WebUI
workspace loading behavior introduced in v2.2.0.

## Fixed

- Configuration Inventory now accepts both legacy numeric route priorities and
  the semantic `critical`, `high`, `normal`, `low`, and `lowest` values written
  by the v2.2.0 WebUI.
- Disabling, enabling, editing, or reordering a route no longer causes
  `/api/v2/configuration/inventory` to return HTTP 400.
- Workspace APIs load independently. If one component fails, successful
  sections remain visible and the failed component is listed in an on-screen
  operational notice.
- Refreshing after a non-authentication API failure no longer signs out a valid
  administrator session.

## Upgrade from v2.2.0

1. Keep the existing `config.yaml`, secrets, logs, and platform-state mounts.
2. Create the normal offline configuration and state backup.
3. Change only the image from `fortpt/notifinho:2.2.0` to
   `fortpt/notifinho:2.2.1`, re-pull, and redeploy.
4. Confirm version 2.2.1 and platform schema 5.
5. Toggle one route, refresh the browser, and confirm the route retains its
   chosen state while every WebUI section remains populated.

No database or YAML migration is introduced by this hotfix. Existing semantic
priorities in `config.yaml` are valid and do not need to be changed back to
numbers.

## Compatibility boundary

- Platform schema remains 5 and the unified YAML model is unchanged.
- Numeric route priorities remain accepted and map to the same semantic names.
- The hotfix does not modify routing filters, destination credentials, notices,
  delivery history, backup schedules, or account data.

## Rollback

Because v2.2.1 does not change schema 5 or the configuration model, a direct
image rollback to v2.2.0 is structurally compatible. Preserve a complete
configuration and state backup before rollout, as for every production change.

## Hotfix acceptance

- Configuration Inventory returns HTTP 200 with `priority: normal` in YAML.
- Disabling and re-enabling a route survives browser refresh.
- A synthetic failure of one workspace endpoint leaves other sections visible
  and identifies the failed component on screen.
- A non-authentication workspace failure does not clear the active session.
- One real notification still produces exactly one matching delivery.
