# Notifinho v1.8.1 release notes

## Release summary

Notifinho v1.8.1 is a presentation and safety patch for every existing source.
It gives Discord embeds and Microsoft Teams Adaptive Cards one shared visual
contract without changing parser routes, endpoints, YAML keys, or transports.

## Presentation contract

- Visible timestamps use `DD Mon YYYY • HH:MM`.
- An explicit `UTC` or numeric offset is retained when the source supplies a
  timezone.
- Discord embeds show a compact product badge in the top-right corner.
- Teams cards use the same title/status treatment with a compact top-right
  product badge.
- Existing field icons, severity colors, links, budgets, and footer metadata
  remain available.
- Xen Orchestra keeps its existing Vates branding and serves as the visual
  baseline.

The contract covers Xen Orchestra, Zabbix, QNAP QTS/QuTS hero, Grafana,
TrueNAS, UniFi Network/Protect/Drive, Portainer, Proxmox VE, and Synology DSM.

## TrueNAS polish

TrueNAS HTML lists are normalized before alert parsing, so wrapped details are
kept with the alert that introduced them. Identical `New` and `Current` copies
of one active condition are presented once, preferring the `New` state.
Cleared/recovery events remain distinct.

## Outbound safety

Every Discord and Teams payload receives a final recursive credential scrub
immediately before delivery. The scrub covers common token, API key, password,
secret, session ID, authorization bearer, Discord webhook, and sensitive query
values. This is defense in depth in addition to parser-specific privacy rules.

## Compatibility evidence

- Portainer BE 2.42.0 firing and resolved events reached dedicated Discord and
  Teams routes through the authenticated native channel.
- A Synology RS822+ running DSM 7.3.2 delivered its custom JSON webhook test
  and a real successful backup event to Teams.
- A Synology DS218+ running DSM 7.1.1 delivered its test notification through
  SMTP with optional STARTTLS and no authentication to Teams.
- A real TrueNAS restart delivered a grouped critical alert through SMTP to
  Discord.
- Proxmox VE remains fixture-validated pending access to a real system.

## Upgrade from v1.8.0

1. Back up the Portainer stack definition and `config/config.yaml`.
2. Run the complete v1.8.1 test and isolated-container checklist from the
   transfer package.
3. Pull `fortpt/notifinho:1.8.1` or
   `ghcr.io/fortpt/notifinho:1.8.1` after the release workflow succeeds.
4. Change only the production image tag from `1.8.0` to `1.8.1`.
5. Preserve all ports, mounts, certificates, secrets, routes, and destination
   webhooks.
6. Recreate the container and confirm the banner reports `1.8.1`.
7. Replay safe fixtures to dedicated development destinations and visually
   inspect timestamps, badges, field icons, colors, and redaction.
8. Monitor production logs and one real source event before completing the
   rollout.

## Rollback

Restore the previous stack definition or pin the image to
`fortpt/notifinho:1.8.0` (or the equivalent GHCR image). No configuration
rollback is required because v1.8.1 adds no YAML keys or transport changes.
