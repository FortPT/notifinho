# Notifinho v1.7.0 release notes

## Release summary

Notifinho v1.7.0 adds native UniFi Drive Alarm Manager webhooks and improves
Drive presentation in Discord and Microsoft Teams. Network, Protect, and Drive
can use the same authenticated HTTPS entry point and global
`X-Notifinho-Token`. Existing Drive delivered-email parsing remains supported.

## Native UniFi Drive endpoint

```text
POST /unifi/drive
```

It runs through the existing optional HTTP listener alongside
`/unifi/network` and `/unifi/protect`. No additional listener, port, or
per-source secret is required.

## Drive payload and alarm naming

The default Drive payload contains only an `alarm_id` and text such as:

```text
Alarm "Notifinho | Drive - Backup Task Partially Completed" was triggered
```

Drive does not include the underlying condition that Protect includes. Create
one alarm per Drive event and use descriptive names. The readable final segment
becomes the card title and the complete rule remains visible as `Alarm rule`.

## Compatibility

- Existing Network and Protect webhooks are unchanged.
- Existing Drive delivered-email parsing remains supported.
- Existing routing keys, output targets, SMTP, STARTTLS, and SMTP AUTH remain
  compatible.
- The HTTP listener remains disabled by default.
- No production configuration or secret migration is required.

## Upgrade from v1.6.0

1. Back up `config/config.yaml` and the current container definition.
2. Pull `fortpt/notifinho:1.7.0` or the equivalent GHCR image.
3. Recreate the container and confirm version `1.7.0`.
4. Confirm the HTTP listener is enabled and mapped.
5. Configure Drive to POST to `/unifi/drive`.
6. Add the same `X-Notifinho-Token` used by Network and Protect.
7. Point the reverse-proxy Drive path to the production Notifinho HTTP listener.
8. Trigger one Drive alarm and confirm HTTP 204 and output delivery.
9. Keep one descriptive Drive event per Alarm Manager rule.

## Rollback

Recreate the container using `fortpt/notifinho:1.6.0`. Network and Protect
continue to work. Native `/unifi/drive` returns 404 on v1.6.0, so restore Drive
email delivery or temporarily route Drive to a compatible listener.
