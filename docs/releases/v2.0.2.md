# Notifinho v2.0.2 release notes

## Release summary

Notifinho v2.0.2 closes the upgrade gap between the existing `config.yaml`
pipeline and the v2 WebUI. An administrator now sees the mounted YAML inputs,
Discord/Teams destinations, routes, filters, enabled state, credential presence,
and active management authority immediately after sign-in. Secret values never
leave the server.

The new mounted-configuration takeover is previewed, fingerprint-bound,
explicitly confirmed, automatically backed up, and single-authority. Once
applied, existing SMTP, Redfish, Home Assistant, and native webhook events use
the destinations and routes managed in the WebUI. The original YAML outputs and
routes remain intact as an inactive rollback fallback.

## Highlights

- Credential-free server-side inventory of the mounted `config.yaml`.
- YAML resources appear beside database resources with clear **YAML managed**,
  **WebUI managed**, **Active authority**, and **Rollback fallback** labels.
- Dashboard destination and route counts follow the authority that handles
  existing sources instead of showing an empty platform database.
- The preview accepts supported Discord and Teams targets, routes, and host
  filters while redacting every credential value.
- Apply requires the unchanged SHA-256 preview fingerprint and administrator
  CSRF confirmation.
- An automatic state backup and atomic configuration backup are created before
  authority changes.
- Credentials move directly from mounted YAML into owner-only secret files;
  they never enter the browser response, preview, audit record, or SQLite file.
- `platform.routing_authority` selects exactly one legacy-event backend:
  `yaml` or `database`.
- A confirmed WebUI control returns delivery immediately to the YAML fallback
  or reactivates database routing without repeating migration.
- Interrupted configuration activation removes only newly imported database
  resources and leaves YAML delivery active.
- Manual YAML upload remains available for another server and stays additive.
- CI and Docker publication use the current Node.js 24 action majors, removing
  the deprecation annotation observed during the v2.0.1 release workflow.

## Fresh installation

1. Deploy `fortpt/notifinho:2.0.2` with persistent configuration, logs,
   secrets, and state mounts.
2. Keep `platform.secure_cookies: true` and publish the WebUI only through a
   trusted HTTPS reverse proxy.
3. Use the single-use setup token from container output to create the first
   administrator.
4. New installations may create destinations and routes directly in the
   WebUI. Keep `platform.routing_authority: database` only after at least one
   valid database route exists.
5. Installations starting from a populated YAML example can use the same
   previewed takeover described below.

## Upgrade from v2.0.1 or v1.9.7

1. Preserve the current stack/Compose definition, mounted `config.yaml`, TLS
   material, logs, and complete platform state directory.
2. Validate the published `2.0.2` image and digest on a development host.
3. Change only the production image tag and recreate the container.
4. Confirm version `2.0.2`, schema 3, both listeners, STARTTLS when configured,
   graceful restart, and normal WebUI login.
5. Before takeover, trigger one real source event and confirm YAML delivery is
   unchanged.
6. Open **Data tools**. Confirm that the mounted inventory matches the active
   inputs, destination targets, routes, and host filters.
7. Choose **Preview safe takeover**. Review all warnings and counts. Apply only
   when the preview is valid and complete.
8. Record the returned state-backup and configuration-backup identifiers.
9. Confirm that the dashboard reports WebUI authority, the imported resources
   are enabled, and the YAML resources are labelled rollback fallback.
10. Trigger exactly one real source event. Verify one destination message, one
    database delivery record, and no duplicate YAML delivery.

The image upgrade alone does not activate database routing. Existing YAML
routing remains authoritative until step 7 is explicitly confirmed.

## Compatibility boundary

The SQLite schema remains schema 3, unchanged from v2.0.1. No database downgrade
or state-directory replacement is required. Existing accounts, sessions,
application tokens, destinations, routes, audit events, delivery history,
backups, and setup-token state remain compatible.

SMTP/HTTP listeners, source authentication, parsers, formatters, suppression,
deduplication, presentation timezone, and integration-specific configuration
remain YAML-managed. Takeover moves only supported Discord/Teams destination
credentials and routes. Unsupported or placeholder targets are reported and
not guessed.

`/api/v2/events` remains scoped to the submitting user or application token.
`platform.routing_authority` affects only legacy SMTP and source-specific
webhook events. Manual YAML-file import does not change authority.

## Security model

- Every mounted-configuration endpoint requires an administrator session;
  state-changing requests also require the session CSRF token.
- The server fingerprints the exact mounted bytes and rejects a changed file
  between preview and apply.
- Inventory and preview expose only names, safe settings, filters, counts, and
  boolean credential presence.
- Secret URLs are validated and stored through the existing owner-only secret
  store with mode `0600` files and digest-only metadata.
- Name collisions, invalid routes, missing targets, unsupported filters,
  repeated takeover, empty migration scope, and invalid authority values fail
  closed.
- Exactly one backend routes each legacy event. Authority flips only after all
  destinations, secrets, and routes exist and the YAML update succeeds.
- Configuration replacement is atomic and creates a mode-`0600` backup. The
  automatic state backup remains integrity-manifested and server-private.
- Authority rollback requires the exact confirmation phrase and creates a new
  configuration backup.

## Rollback

### Routing rollback without image downgrade

Use **Data tools → Use YAML fallback**. Confirm the operation, then send one
real test event. Notifinho changes only `platform.routing_authority` to `yaml`,
creates another configuration backup, and immediately resumes the retained YAML
routes. Database resources and credentials remain intact for investigation or
later reactivation.

### Image rollback to v2.0.1

First return routing authority to YAML and verify delivery. Then pin the
previously validated `2.0.1` image and recreate the container. Because both
releases use schema 3, the v2.0.2 database can remain mounted. v2.0.1 ignores
the new authority setting and continues its YAML legacy pipeline; imported
platform resources remain available only to platform event submissions.

Keep the automatic pre-takeover state backup and YAML backup until the rollback
is accepted. Restore the complete host-level backup only when state itself is
damaged. Never partially copy the SQLite file and secret directory.

## Release acceptance

- Python compilation, WebUI JavaScript syntax, YAML parsing, Compose
  validation, and the complete automated test suite pass.
- Inventory tests prove that webhook URLs and shared secrets never enter public
  API JSON.
- Migration tests cover preview fingerprints, automatic state backup,
  configuration backup, secret import, authority activation, repeated-import
  rejection, rollback confirmation, and cleanup after a failed YAML switch.
- Runtime routing tests prove that a legacy parsed notification reaches exactly
  one imported database route after takeover and records delivery history.
- Browser contract tests cover mounted inventory, active/fallback labels,
  takeover controls, same-origin requests, CSRF, and unsafe-DOM exclusions.
- An isolated container upgrade from a populated v2.0.1 YAML configuration
  verifies pre-takeover YAML delivery, browser preview/apply, schema 3
  persistence, one real post-takeover delivery, authority fallback/reactivation,
  and graceful shutdown.
- Pull-request and merged-main CI pass before tag `v2.0.2` is pushed.
- The tag workflow publishes matching Docker Hub and GHCR `2.0.2` and `latest`
  images and creates the GitHub Release from this document.
