# Notifinho v2.0.0 release notes

## Release summary

Notifinho v2.0.0 turns the existing parser-driven notification engine into an
opt-in, self-hosted notification platform. It adds local administrator and user
accounts, source-scoped application tokens, owned destinations and routes,
six platform output adapters, an authenticated API and WebUI, safe operational
history, credential-free portability, previewed v1.x migration, and private
state backup and restore.

The existing SMTP and native webhook pipeline remains available and unchanged.
Platform state, the authenticated API, and the WebUI are disabled unless an
operator explicitly enables each layer.

## Highlights

- Migration-aware SQLite state with local roles, login lockout, hashed sessions
  and CSRF credentials, last-administrator protection, and owner-only secret
  rotation.
- User- and application-scoped event submission with revocable, source-scoped
  tokens and owner-filtered routing.
- Private and shared destinations plus source, host, event, severity, and status
  route filters.
- Discord, Microsoft Teams, Slack, generic HTTP webhook, MQTT, and ntfy
  platform adapters using backend previews and authorized test delivery.
- Responsive same-origin WebUI for account, token, destination, route,
  delivery-history, audit, and administrator workflows.
- Credential-free JSON export/import with merge-only semantics and exact
  preview fingerprints.
- Previewed Discord and Teams v1.x YAML migration with write-only credential
  handling.
- Integrity-manifested server-side state backups with private permissions,
  bounded retention, safety snapshots, staged restore rollback, and browser
  session revocation.
- Production Compose hardening, pinned dependencies and base image, JavaScript
  syntax validation, production Compose validation, and production-image CI.

## Compatibility boundary

Existing YAML inputs, API tokens, routing, Discord and Teams targets, parsers,
formatters, ports, and environment variables continue to work. An existing
configuration that omits `platform.enabled` and `webui.enabled` retains the
v1.9.7 runtime behavior.

The platform database does not replace the YAML router. Administrators can
preview and import supported Discord and Teams targets and routes, but the
migration never rewrites the YAML file. Running both route models can send
duplicate notifications, so move traffic deliberately.

Exports exclude passwords, session material, API-token values and hashes,
secret identifiers and digests, webhook URLs, and destination passwords.
State backups remain inside the private state mount and are not downloadable
through the API or WebUI.

## Upgrade from v1.9.7

1. Back up the production Compose definition, `.env`, configuration, secrets,
   logs, and any existing state directory.
2. Validate `fortpt/notifinho:2.0.0` or
   `ghcr.io/fortpt/notifinho:2.0.0` on the development VM before production.
3. Change only `NOTIFINHO_IMAGE` and recreate the container. Leave the new
   platform and WebUI switches disabled for the initial compatibility check.
4. Confirm SMTP and any enabled native HTTP inputs start, existing routes still
   deliver, the container remains non-root, and shutdown is graceful.
5. To adopt the platform, mount a private writable state directory, configure
   `http.enabled`, `api.enabled`, and `platform.enabled`, then bootstrap the
   first administrator from the trusted container shell:

   ```bash
   docker compose -f compose.production.yaml exec notifinho \
     python3 tools/manage_users.py create-admin --username administrator
   ```

6. Put the HTTP listener behind HTTPS before enabling `webui.enabled`; retain
   secure cookies, request limits, and the documented reverse-proxy headers.
7. Create a private state backup before importing v1.x routes or moving live
   delivery to platform destinations.

## Rollback

If only the legacy YAML pipeline is active, set `NOTIFINHO_IMAGE` back to the
previously validated v1.9.7 image and recreate the container with the same
configuration and secret mounts.

If platform state was enabled, first stop writes and create or preserve an
off-host copy of the complete state directory. Disable `platform.enabled` and
`webui.enabled` before starting v1.9.7; the older image ignores the preserved
platform state. Do not delete or reverse-migrate the v2 database. Return to
v2.0.0 to restore platform operation.

## Release acceptance

- Complete Python suite passes on Python 3.13 with pinned dependencies.
- Python source compilation and WebUI JavaScript syntax checks pass.
- Public YAML and both Compose definitions validate.
- The production image builds from the immutable Python base and contains the
  WebUI, platform storage, migration, backup, and user-management assets.
- An isolated non-root, read-only VM-09 runtime passes login/CSRF, ownership,
  six-adapter preview/test, scoped event delivery, portability, migration,
  backup/restore, session revocation, permissions, and graceful shutdown.
- The existing VM-09 development configuration starts unchanged with platform
  features disabled.
- A manual browser pass confirms responsive login, navigation, destination and
  route workflows, preview, history, Data tools, logout, and no stored secret
  disclosure.
- VM-04 production remained unchanged throughout release acceptance; production
  rollout is a separate, explicitly approved and validated operation.
- The signed-off `v2.0.0` tag published matching Docker Hub
  and GHCR images and created the GitHub Release from this document.
