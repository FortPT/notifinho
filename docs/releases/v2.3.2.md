# Notifinho v2.3.2 release notes

Released 23 July 2026.

v2.3.2 is a corrective release for source identity and lifecycle, destination
tests, HTTP/HTTPS session migration, restart placement, and directly managed
NFS/SMB backup destinations. It retains the unified `config.yaml` model and
platform schema 6.

## Highlights

- Overview Routing Flow and Sources use the official vendor assets already
  packaged with Notifinho. Custom, wildcard, and otherwise unknown sources use
  the Notifinho icon.
- Source categories are now Virtualization, Monitoring, Storage, Networking,
  Hardware, Automation, Containers, Security, and Generic. Stored v2.3.1 tags
  are mapped safely during upgrade.
- An enabled `*` route marks every discovered source Active. Administrators can
  remove only inactive sources, after confirmation. Removal hides the source
  and clears its saved category without deleting delivery history.
- Destination-card tests use a generic Notifinho event containing the selected
  destination name and output type. They no longer display Home Assistant
  branding.
- Restart moves from Settings to the top-right header. It remains
  administrator-only, CSRF-protected, reason-required, and audited.
- Session lookup prefers the cookie corresponding to
  `platform.secure_cookies`. An old `__Host-notifinho_session` cookie can no
  longer override the current standard cookie when HTTP and HTTPS are both
  enabled.
- The managed-backup Compose override adds `DAC_OVERRIDE`, `FOWNER`, and
  `SYS_ADMIN`. This is the minimum tested set required for root to access
  existing UID-owned bind mounts, apply the platform state mode, and mount a
  remote filesystem after the base service drops all capabilities.
- Managed NFS mounts include `nolock`. Backup archive writes do not depend on
  advisory locks, and NFSv3 therefore does not need `rpc.statd` to write below
  `/run` on the read-only container filesystem.

## Direct managed-mount deployment

The normal production service remains non-root and capability-free. Apply the
managed override only when Notifinho must mount NFS/SMB itself:

```bash
docker compose \
  -f compose.production.yaml \
  -f compose.managed-backups.yaml \
  config

docker compose \
  -f compose.production.yaml \
  -f compose.managed-backups.yaml \
  up -d
```

The merged service has:

```yaml
user: "0:0"
cap_drop:
  - ALL
cap_add:
  - DAC_OVERRIDE
  - FOWNER
  - SYS_ADMIN
security_opt:
  - no-new-privileges:true
```

Do not add `privileged: true`. Restrict the NFS/SMB export to the Notifinho host
and the dedicated backup path.

For the validated UNAS arrangement:

```text
Host: 192.168.0.163
NFS export: /var/nfs/shared/Backups/Notifinho
Allowed client: 192.168.0.164
Access: Read/Write
```

`vers=3` remains an optional target mount option when required by the server.
Notifinho adds `nolock` internally.

## HTTP and HTTPS access

Both endpoints can remain available with:

```yaml
platform:
  secure_cookies: false

webui:
  public_url: ""
  enforce_https: false
```

The IP address and HTTPS hostname have independent browser cookie stores and
therefore require separate logins. v2.3.2 handles legacy and current cookie
names in the correct configured order.

## Upgrade

1. Stop Notifinho and copy the complete `config`, `state`, `logs`, and secrets
   mounts.
2. Pull `fortpt/notifinho:2.3.2`.
3. When using direct NFS/SMB mounting, deploy both Compose files above.
4. Start Notifinho.
5. Complete the
   [v2.3.2 acceptance checklist](../v2.3.2-acceptance-checklist.md).

## Compatibility and rollback

v2.3.2 does not change schema 6. A direct image rollback to v2.3.1 is supported
after stopping the container, provided the configuration and state are kept as
one matched set. Source removal adds only the optional
`webui.removed_sources` list; v2.3.1 ignores that key at runtime but its older
schema validator does not accept it, so restore the pre-upgrade configuration
before a deliberate rollback when the removal feature was used.
