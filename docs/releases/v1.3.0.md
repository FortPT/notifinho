# Notifinho v1.3.0 release notes

## Highlights

### QNAP QTS and QuTS hero

Notifinho adds provisional source detection, parsing, and source-specific
Discord and Microsoft Teams formatting for common QNAP notification groups:

- storage pools, RAID, disks, and SMART warnings;
- failed logins and security events;
- HBS and backup failures;
- firmware and application notices; and
- UPS and power events.

See the [QNAP integration guide](../qnap.md) for configuration, fixture replay,
and current limitations.

### Grafana Alerting

Notifinho adds provisional support for Grafana test notifications, firing and
resolved alerts, pending and No Data states, datasource or evaluation errors,
and grouped alerts. Grafana notifications receive dedicated Discord embeds and
Microsoft Teams Adaptive Cards.

See the [Grafana integration guide](../grafana.md) for configuration, fixture
replay, and current limitations.

### SMTP fixture replay

The reusable `scripts/replay_email.py` utility sends an `.eml` fixture to an
SMTP server without authentication. Development defaults are `127.0.0.1:8026`:

```bash
python3 scripts/replay_email.py \
  tests/fixtures/qnap/storage_warning.eml \
  --host 127.0.0.1 \
  --port 8026

python3 scripts/replay_email.py \
  tests/fixtures/grafana/alert_firing.eml \
  --host 127.0.0.1 \
  --port 8026
```

### Dedicated routing targets

QNAP and Grafana can use independent webhook targets while retaining the
default destination for other sources:

```yaml
outputs:
  discord:
    default:
      webhook: "PASTE_DISCORD_WEBHOOK_HERE"
    qnap:
      webhook: "PASTE_QNAP_DISCORD_WEBHOOK_HERE"
    grafana:
      webhook: "PASTE_GRAFANA_DISCORD_WEBHOOK_HERE"

routing:
  qnap:
    outputs:
      - output: discord
        target: qnap
  grafana:
    outputs:
      - output: discord
        target: grafana
```

Equivalent Microsoft Teams targets can be configured under `outputs.teams`.
The public [`config/config.example.yaml`](../../config/config.example.yaml)
contains the complete examples. Existing private `config/config.yaml` files
should be updated locally and must not be committed.

## Release automation

Stable `vMAJOR.MINOR.PATCH` tags trigger the release workflow. The workflow:

- validates the stable tag format;
- verifies that the checked-out commit belongs to the requested tag;
- verifies that `src/version.py` matches the tag;
- publishes versioned and `latest` images to Docker Hub and GHCR;
- creates the corresponding GitHub Release;
- updates the existing release instead of creating a duplicate during reruns;
  and
- supports manual publication of an existing stable tag.

Release notes are loaded from `docs/releases/vX.Y.Z.md` when that file exists.
Generated GitHub notes are used as a fallback.

## Test coverage

The release suite contains 123 passing tests covering source detection,
plain-text/HTML/multipart parsing, status normalization, Discord and Teams
payloads, output formatter selection, routing regressions, malformed data, and
Discord aggregate payload budgets.

## Upgrade notes

- The application version is now `1.3.0`.
- Compare new public configuration examples with the deployment's private
  `config/config.yaml`; do not replace real webhook values with placeholders.
- Add `routing.qnap` or `routing.grafana` only when dedicated destinations are
  wanted. Existing Xen Orchestra, Zabbix, default, and conditional routes
  remain supported.
- Development SMTP listens on host port `8026`; production SMTP continues to
  use port `8025`.

## Provisional compatibility

All current QNAP and Grafana fixtures are synthetic. They exercise the initial
formats but do not prove compatibility with every QTS, QuTS hero, Grafana
version, or customized alert template. Anonymized real `.eml` samples are
requested for compatibility hardening after the provisional integration.
