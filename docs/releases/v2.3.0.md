# Notifinho v2.3.0 release notes

Released 23 July 2026.

v2.3.0 completes the operational WebUI work requested after v2.2.1. It keeps
the unified `config.yaml` routing model and upgrades private platform state to
schema 6.

## Highlights

- A shorter, centered login surface with a larger brand and optional HTTP to
  canonical HTTPS redirect through `webui.public_url`.
- Immediate notice, health-check, Audit Log, destination-test, and backup-run
  updates without requiring F5.
- Notice editing/resolution and first-login enrollment: every new account sees
  the two platform defaults, then receives announcements published after its
  first login.
- Categorized Overview sources, real HTTP/SMTP input labels, user read-only
  metrics and activity, and animated route direction with reduced-motion
  support.
- Channel-aware destination cards, one-click card tests, edited-preview tests,
  immutable output types, and priority-only route ordering.
- Semantic Delivery History with device/event titles, readable source and
  input labels, a dedicated description panel, and red error/failure state
  independent of successful transport delivery.
- Audit Log page sizes from 25 through 500, circular avatar crop/zoom, header
  Sign out, and an administrator-only reasoned restart action.
- Separate Inputs and Backups views with named Local, NFS, and SMB backup
  destinations, masked credentials, connectivity/write tests, manual runs,
  schedules, last result, and next-run presentation.

## Upgrade

1. Stop Notifinho and copy the complete configuration and state directories.
2. Pull `fortpt/notifinho:2.3.0` in the normal hardened Compose deployment.
3. Start the service and sign in over HTTPS.
4. Follow the [v2.3.0 acceptance checklist](../v2.3.0-acceptance-checklist.md).

Startup creates a pre-migration SQLite snapshot before installing schema 6.
The migration adds first-login notice enrollment metadata, mutable notice
timestamps, and the backup-destination catalog. Existing YAML inputs,
destinations, routes, priorities, state backups, users, and delivery history
remain compatible.

To redirect a plain HTTP login URL, configure the externally reachable TLS URL:

```yaml
webui:
  enabled: true
  public_url: "https://notifinho.example.com"
```

Notifinho issues a 308 redirect; the reverse proxy must still provide the HTTPS
listener and certificate.

## Backup destination security

The recommended production model remains mounting an NFS or SMB share on the
Docker host and exposing a bounded local path to Notifinho. This preserves the
non-root user, dropped capabilities, and host credential boundary.

Operators who specifically require Xen-Orchestra-style application-managed
mounts may enable `platform.backups.managed_mounts` and layer
`compose.managed-backups.yaml` over production Compose. That override runs the
container as root and grants `SYS_ADMIN`; use it only on a dedicated trusted
host. SMB passwords are encrypted in owner-only state, are write-only in the
API, and are never displayed again.

## Restart behavior

Restart requests require an administrator session, CSRF token, confirmation,
and a reason of 3–500 characters. The reason is recorded in Audit Log before
Notifinho exits. A production supervisor such as Docker with
`restart: unless-stopped` brings the container back; this is not a host reboot
and no shutdown endpoint is provided.

## Rollback

Schema 6 cannot be opened by v2.2.1. To roll back, stop Notifinho, restore the
complete pre-v2.3.0 configuration and state directories, pin
`fortpt/notifinho:2.2.1`, and start the stack. Do not edit the schema version or
copy only the SQLite file without its secret directory.
