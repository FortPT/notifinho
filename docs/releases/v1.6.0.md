# Notifinho v1.6.0 release notes

## Release summary

Notifinho v1.6.0 adds optional SMTP transport security using explicit STARTTLS
and SMTP authentication.

SMTP security remains disabled by default, so existing deployments continue to
operate without configuration changes.

## Security features

- Explicit STARTTLS support
- TLS 1.2 minimum
- SMTP AUTH LOGIN and PLAIN
- AUTH unavailable before TLS
- Timing-safe username and password comparisons
- Password loading from an environment variable or secret file
- Fail-closed startup validation
- Secret-safe logging
- Secure enablement defaults
- LOGIN and PLAIN disabled in TLS-only mode
- Submitted credential data is not retained after authentication

Implicit SMTPS is not supported. The listener continues to use SMTP port
`8025`, with STARTTLS negotiated on the existing connection.

## Backward-compatible default

An existing configuration remains valid:

```yaml
smtp:
  host: "0.0.0.0"
  port: 8025
```

This keeps the listener in its original plaintext and unauthenticated mode.

Firewall source restrictions remain strongly recommended even when TLS and
authentication are enabled.

## STARTTLS and authentication

```yaml
smtp:
  host: "0.0.0.0"
  port: 8025

  tls:
    enabled: true
    certfile: "/notifinho/config/tls/cert.pem"
    keyfile: "/notifinho/config/tls/key.pem"

  auth:
    enabled: true
    username: "notifinho"
    password_env: "NOTIFINHO_SMTP_PASSWORD"
    password_file: ""
```

When `tls.enabled` is true, STARTTLS is required by default.

When `auth.enabled` is true, authentication is required by default.

The password is not stored directly in `config.yaml`.

## Environment-variable secret

Example Compose configuration:

```yaml
services:
  notifinho:
    image: fortpt/notifinho:1.6.0

    environment:
      NOTIFINHO_SMTP_PASSWORD: "${NOTIFINHO_SMTP_PASSWORD}"

    volumes:
      - ./config:/notifinho/config
      - ./config/tls:/notifinho/config/tls:ro
      - ./logs:/notifinho/logs
```

Set the variable through the deployment platform rather than committing it to
the repository.

## Docker secret file

Configure:

```yaml
smtp:
  auth:
    enabled: true
    username: "notifinho"
    password_env: ""
    password_file: "/run/secrets/notifinho_smtp_password"
```

Mount the secret at the configured path. Exactly one password source must be
usable.

## Certificates

The certificate must:

- match the hostname used by SMTP senders;
- contain the necessary intermediate certificate chain;
- be readable by the Notifinho container;
- have a private key matching the certificate;
- be trusted by each sending appliance.

The certificate and key under `tests/fixtures/tls` are synthetic test fixtures
and must never be used in a deployment.

## Upgrade

1. Back up `config/config.yaml`.
2. Pull the `1.6.0` image.
3. Recreate the Notifinho container.
4. Confirm that the application reports version `1.6.0`.
5. Leave SMTP security disabled for the initial compatibility check.
6. Mount the production certificate and private key.
7. Configure the password environment variable or secret file.
8. Enable TLS.
9. Enable authentication.
10. Test every sending appliance separately.
11. Retain firewall source allowlisting.

## Rollback

Recreate the container using `fortpt/notifinho:1.5.2` or the equivalent GHCR
image.

Existing plaintext configurations remain compatible, so no configuration
rollback is required unless SMTP security settings were added.

## Compatibility notes

Each sender must support:

- explicit STARTTLS;
- TLS 1.2 or newer;
- SMTP AUTH LOGIN or PLAIN;
- certificate hostname validation and trust appropriate to its implementation.

Some infrastructure products support STARTTLS without SMTP authentication, or
authentication without custom certificate trust. Validate appliance support
before making STARTTLS or authentication mandatory in production.
